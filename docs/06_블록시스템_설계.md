# 06. 조립 분석 — 블록 시스템 아키텍처 설계

> 목적: `/lab`의 "기업 1개 → 지표 카드" 구조를, **여러 기업을 끌어와 비교·연산·필터·표시**하는 진짜 데이터플로우 그래프로 진화시키기 위한 설계 청사진.
> 사용자 요구의 핵심 문장: *"기업 블록도 가져오고… 여러 기업이 연결될 시 회사명 +n%… 블록의 속성… 번들… 디스플레이 블록… 분석 기법에 맞게끔."*

---

## 0. 한 줄 결론

개인투자자 재무 분석 기법(밸류에이션·수익성·성장·건전성·peer 상대가치·스코어링·시계열추세·스크리닝)은 모두 **7개 블록 프리미티브의 조합**으로 환원된다. 포트(소켓)에 **타입**을 붙여 "되는 연결만" 강제하고, 멀티-기업은 **broadcast(기업별 행) vs aggregate(집계)** 두 시맨틱으로 나누면, 과한 추상화 없이 현재 코드의 **자연스러운 일반화**가 된다.

(근거 조사 도구: YCharts·Stock Rover의 가중 스코어링, Piotroski F-score / Magic Formula = 이미 이 파이프라인을 구현. 노드시스템 선례: Blender Geometry Nodes, Grasshopper, Unreal Blueprint, Houdini, n8n, Node-RED, Enso.)

---

## 1. 현재 구조 진단

지금 `/lab`은 노드 에디터가 아니라 **별(star) 그래프**다.

- 페이지당 `company` 노드 **1개(고정 id `"company"`)** + `metric` 말단 카드 N개.
- 엣지는 평가에 안 쓰임 — 단지 `company.companyId`를 `metric`에 주입하는 **1-hop 트리거**.
  (`onConnect`: `source !== "company"`면 거부, 타깃당 입력 1개만.)
- 평가 = 노드별 `block.compute(company)` **1단계 호출**. DAG 위상정렬·다단계 연산 없음.

→ 사용자가 원하는 진화(다수 기업 + 다-기업 fan-in + 번들/디스플레이/그룹 + 분석기법 블록)는 **"값을 흘려보내는 진짜 데이터플로우 DAG"**를 요구한다. 그래서 먼저 ①포트에 타입을 붙이고 ②노드를 카테고리로 분류하고 ③리스트(다수 기업) fan-in 시맨틱을 명시해야 한다.

---

## 2. 블록 카테고리 — 7개 프리미티브로 고정

| # | 카테고리 | 역할 | 예시 블록 |
|---|----------|------|-----------|
| 1 | **Source(소스)** | 그래프의 입력 | 기업 블록, 기업셋(CompanySet), 기간(Period) |
| 2 | **Metric(지표)** | 기업 → 숫자 | ROE, PER, 영업이익률, 매출성장률 |
| 3 | **Operator(연산)** | 숫자 → 숫자 | 사칙연산, 비율(A/B), 증감률 |
| 4 | **Compare/Aggregate(비교·집계)** | 리스트 → 요약 | 평균, 순위(Rank), 최대/최소, 비교 |
| 5 | **Filter(필터)** | 조건으로 추림 | "ROE>15%", "PER<20" → 통과 기업셋 |
| 6 | **Display(표시)** | 결론 sink | 표(Table), 도넛, 스코어카드, **Verdict(결론)** |
| 7 | **Group(그룹)** | 시각적/논리적 묶음 | **번들(Bundle)**, 그룹 박스 |

> 카테고리를 7개로 **고정**하는 게 핵심. 무한히 늘리면 입문자 학습 비용이 폭발한다(THESIS의 "입문자도 쉽게"에 정면 위배).

---

## 3. 포트(소켓) 타입 — "되는 연결만 보이게"

### 타입 집합 (MVP는 4~5개로 한정)

| 타입 | 색(예시) | 의미 |
|------|----------|------|
| `Company` | sky | 기업 1개 |
| `CompanySet` | indigo | 기업 N개(리스트) |
| `Number` | amber | 스칼라 값(지표 결과) |
| `NumberList` / `Table` | violet | 기업별 값들(행 모음) |
| `Boolean` *(나중)* | teal | 필터 통과 여부 |

> Blender(소켓 모양)·Grasshopper(데이터 트리)·Enso(타입추론 색) 모두 **포트 타입을 색·모양으로 시각화해 오연결을 막는다**. 단 풀(full) 데이터트리·도메인 자동변환은 MVP에 과함 → **4~5타입으로 좁힌다.**

### React Flow에서의 강제 방법
- React Flow에는 **빌트인 `maxConnections`가 없다** → `isValidConnection` + handle id에 타입 인코딩으로 직접 검증.
- handle id 규약: `"in:Company"`, `"out:Number"` 식으로 타입을 박아 둠.
- `isValidConnection(conn)` = `sourceType`이 `targetType`에 호환될 때만 true. 호환 안 되면 **연결선이 아예 안 생김**(빨간 X도 없이 깔끔히 거부).
- fan-in 규칙(입력당 몇 개 허용)도 `onConnect`/`isValidConnection`에서 직접 시행.

---

## 4. 멀티-기업 fan-in 시맨틱 ★ 가장 중요

사용자의 *"여러 기업이 연결되면 회사명 +n%"* = 정확히 **broadcast** 시맨틱이다.

| 시맨틱 | 동작 | 출력 | 비고 |
|--------|------|------|------|
| **Broadcast(기본)** | 지표/연산 블록이 Company N개를 받아 **기업별로 반복 적용** | 기업별 행: `삼성전자 12.4% / SK하이닉스 9.1% …` | 기본값. 사용자가 원한 "회사명 +n%" |
| **Aggregate(명시)** | 리스트를 **하나로 reduce** | 평균/순위/최대 등 스칼라·테이블 | 별도 블록(평균·Rank)으로만 |

> Grasshopper가 가장 중요하게 다루는 의미론도 이것 — "리스트를 항목별 반복할지 vs 하나로 집계할지". **broadcast가 기본, aggregate는 명시 블록**으로 두면 추상화 폭주 없이 직관적이다.
> 길이 다른 두 리스트 짝짓기(Longest/Shortest/Cross-Reference)는 MVP에서 **보류** — 우선 "지표 1개 + 다수 기업"의 1차원 fan-in만 지원.

## 5. 번들 / 디스플레이 / 그룹

| 개념 | 정의 | 구현 |
|------|------|------|
| **번들(Bundle)** | 같은 Company 입력을 여러 블록이 **공유**하게 묶는 패스스루 그룹 | "이 기업으로 ROE·PER·성장률 한 번에" — 입력 1개를 fan-out |
| **디스플레이(Display)** | 여러 Value/Set 출력을 **표/도넛/스코어카드**로 모으는 sink | 결론 먼저 노출. 평가 그래프의 종점 |
| **그룹(Group)** | 노드 여러 개를 시각적으로 접는 박스 | React Flow **부모 노드(parent)** 기능 그대로 |

> 그룹/서브그래프는 **가장 마지막** (또는 보류). 시각 정리용이라 코어 평가와 무관.

## 6. 데이터 모델 (타입, `any` 없이)

```ts
// 포트 타입 — 디스크리미네이티드 유니온
type PortType = "Company" | "CompanySet" | "Number" | "NumberList";

interface Port { id: string; type: PortType; label: string; }

interface BlockDef {
  id: string;                 // "roe", "rank", "table" …
  category: BlockCategory;    // 7종
  inputs: Port[];
  outputs: Port[];
  tier: "free" | "pro" | "quant";
  evaluate(inputs: PortValue[]): PortValue[];   // 현 compute(company) 의 일반화
}

// 평가 결과(브로드캐스트면 기업별 행)
type PortValue =
  | { type: "Company"; company: CompanyData }
  | { type: "CompanySet"; companies: CompanyData[] }
  | { type: "Number"; rows: { company: string; value: number; display: string; tone: Tone }[] }
  | { type: "NumberList"; ... };
```

- 현재 `PageGraph`(persistence.ts)의 `companyId: string` 단일 가정을 제거하고, **노드 배열 + 타입드 엣지**로 일반화.
- jsonb 저장이라 **스키마 추가만으로 마이그레이션 불필요**(v2→v3 normalize로 하위호환).

### 평가 모델
- 현재: per-node `compute(company)` 1패스.
- 목표: **DAG 위상정렬(topo-sort) 1패스 evaluate**. 사이클 금지(포워드 전용), 입력당 단일 소스.

## 7. 추천 스타터 블록셋 (우선순위 + 플랜)

| 블록 | 카테고리 | 플랜 | 비고 |
|------|----------|------|------|
| 기업 블록(다중 인스턴스) | Source | Free | 현재 단일 고정 → 팔레트 드래그형 |
| 기업셋(CompanySet) | Source | Free | 여러 기업 한 번에 |
| ROE / 영업이익률 / 매출성장 / PER | Metric | Free | **현재 4지표 그대로 살림** |
| PEG | Metric | Pro | (기존 pro 플래그) |
| 비율·증감(Operator) | Operator | Free | A/B, 사칙 4종+비율로 **제한** |
| 비교 테이블(Compare→Table) | Compare+Display | Free | "회사명 +n%" 표 |
| 순위(Rank) | Aggregate | Pro | 다기업 정렬 |
| 필터(Screener) | Filter | Pro | "ROE>15%" 추리기 |
| 도넛/스코어카드 | Display | Free | 결론 먼저 |
| **Verdict(결론)** | Display | Free | **"사세요" 금지 가드 강제** |
| 가중 Score | Aggregate/Display | Pro | "백테스트 아님·표본편향" 경고 |
| Custom Formula | Operator | Pro/Quant | 자유 수식 |

## 8. 설계 철학 매핑 (THESIS 7원칙과의 정합)

- **타입드 포트 = 과신 방지 가드**: 의미 없는 연결(값을 기업 슬롯에)을 물리적으로 차단.
- **결론 먼저**: Display·**Verdict** 블록이 결론, 연산 근거는 펼쳐보기.
- **결론 무료 / 근거 유료**: Score·Custom Formula·시계열·다기업 비교를 **Pro 게이트**.
- **과신 방지 문구**: 자작 Score·Filter엔 *"과거 데이터 기반·백테스트 아님·표본 편향 가능"* 경고. Verdict 레벨에서 **"사세요" 금지** 강제.
- **한국 특화**: 기업셋 기본값에 코스피/반도체 등 한국 종목 우선.

## 9. 단계적 구현 순서 (실용주의)

| 단계 | 내용 | 난이도 | 핵심 리팩토링 |
|------|------|--------|----------------|
| **1** ★MVP핵심 | 포트에 타입 부여 + `isValidConnection` 검증 | 낮음 | 현 트리 구조 유지하며 handle에 타입 |
| **2** | 기업 노드를 **팔레트 드래그형 다중 인스턴스**로 | 중 | 고정 `"company"` id 제거 → `onConnect`/serialize/edge id 전반 |
| **3** | 지표블록 **다-기업 fan-in → 회사명 +n% 행 출력** | 중 | `compute(company)` → `evaluate(inputs[])` 전환 |
| **4** | Display/Compare/Aggregate/Filter 도입 | 상 | **진짜 DAG 위상정렬 평가 엔진** |
| (말미/보류) | Group/서브그래프, Custom Formula, Cross-Reference 매칭 | — | 시각 정리·고급 |

---

## 10. 과한 추상화 경계 (하지 말 것)

- ❌ 완전 일반 dataflow 엔진(Grasshopper급 무한 데이터트리) 짓지 말 것.
- ✅ 포트 타입 **4~5개로 한정**.
- ✅ Operator는 **산술 4종 + 비율**로 제한.
- ✅ fan-in은 **broadcast 기본 + aggregate 명시 블록**, 1차원만.
- ✅ Group/서브그래프는 **마지막** 또는 보류.
- ✅ 길이 다른 리스트 매칭(Longest/Cross-Reference)은 **보류**.

## 부록 — 노드시스템 선례 요약

- **Blender Geometry Nodes**: 소켓 *모양*으로 단일값/필드/리스트 구분, 도메인 자동변환.
- **Grasshopper**: 모든 데이터=데이터트리, item/list/tree 자동 반복, Longest/Shortest/Cross-Reference 매칭.
- **Unreal Blueprint**: exec핀(실행순서)과 data핀(값) 분리, autocast, wildcard 핀.
- **Houdini**: Merge로 스트림 합치기(속성 보존), Subnet/HDA로 복잡도 접기.
- **n8n**: Trigger/Action/Transform, Merge에 Append·Combine·Position·SQL 4모드.
- **Node-RED**: Inject(소스)/Debug(싱크)/Function/Change/Switch + Subflow/Link.
- **Enso**: 타입 추론으로 노드·연결 색 자동 결정 → 그래프 한눈에.

> 공통 교훈: (A) 포트에 타입을 붙여 색/모양으로 보여주고 오연결을 막는다. (B) "리스트 입력을 항목별 반복할지 vs 집계할지"가 가장 중요한 의미론이다. → THESIS는 이 둘만 충실히 가져가고 나머지는 버린다.
