# 04. 아이디어 검토 — 블록 조립형 재무분석 노드 에디터

> 검토일: 2026-06-18 · 검토 방식: 5개 렌즈(PMF/타겟, 기술복잡도, 선례리서치, BM, 철학정합+리스크) 적대적 종합 + 코드베이스 직접 검증
> 결론을 먼저, 근거는 아래에. (THESIS 철학3 — 이 문서 자체에도 적용)

## 0. 요약 Verdict — **PIVOT (현 형태 NO-GO)**

**현재 제안된 "입문자도 쓰는 무한 캔버스 노드 에디터"는 NO-GO다.** THESIS의 1순위 타겟(입문자), 핵심 철학(③ 복잡도 숨김·⑥ 과신 방지), 그리고 데이터 현실(재무제표 0% 배선) 세 가지와 동시에 충돌하기 때문이다. 5개 렌즈가 만장일치로 같은 결론에 도달했고, 코드 검증도 이를 뒷받침한다.

그러나 아이디어를 **버릴 필요는 없다.** 동기는 정당하고, 좁히면 살아난다. 그래서 종합 판정은 **PIVOT**:

- **메인 제품 컨셉으로는 반대(NO).** 입문자 흐름(매크로→자산배분→기업분석, 결론 먼저)에 노드 에디터를 일절 노출하지 않는다.
- **격리·후순위화하면 조건부 가능(PIVOT).** ① 데이터 토대(특히 한국 재무제표)를 먼저 확보하고, ② 자유 캔버스가 아니라 "검증된 블록의 제한된 프리셋 조합"으로 변형하고, ③ Quant(₩29,900) 전용 파워레이어로 격리하고, ④ MVP 5단계(스크리너·포트폴리오)와 출시 전 필수(마이그레이션·E2E 인증·법적고지)를 끝낸 뒤 6단계 이후로 미룬다는 4중 조건 하에서만.

한 줄 근거: **노드 에디터는 "기능 1개"가 아니라 "제2의 제품(플랫폼)"이고, 연결할 데이터가 지금 0이며, 가장 비입문자적인 UI를 입문자 타겟 제품의 깔때기로 쓰는 자기모순이다.**

## 1. 이 아이디어는 진짜 누구를 위한 것인가

| 질문 | 솔직한 답 |
|---|---|
| 입문자(Free, 1순위 타겟)를 위한 것인가? | **아니다.** 입문자는 정의상 "ROE=순이익/자본"을 직접 잇고 싶어하지 않는다. 리테일 투자자는 do-it-for-me·단일 결론을 선호한다([finimize](https://finimize.com/business/resources/insights/how-retail-investors-make-decisions)). 이 도메인에서 입문자에게 실제로 통한 승자(Simply Wall St 스노우플레이크)는 조립을 **위임한 게 아니라 제거**했다([Simply Wall St](https://support.simplywall.st/hc/en-us/articles/360001740916-How-does-the-Snowflake-work)). |
| 중급자(Pro)를 위한 것인가? | **부분적으로.** 입력값을 바꿔 시나리오를 돌리고 근거를 깊게 파는 수요는 있다. 단, "근거 펼쳐보기" 수준이면 충분하고 자유 배선까진 과하다. |
| 퀀트(Quant, ₩29,900)를 위한 것인가? | **그렇다 — 유일한 진짜 타겟.** "PEG<1·ROE>15%·OPM>20%" 조합을 직접 저장·재실행하려는 수요는 실재하고(docs/01 §3-5 커스텀 스크리너·§3-8 백테스트가 이미 Quant로 명세), Composer.trade·FinceptTerminal 같은 선례가 PMF를 증명한다. |

**핵심:** 이 아이디어를 원하는 사람은 **소수의 파워유저/퀀트**다. THESIS 수익모델의 다수층(Free 유입→Pro ₩9,900 전환)과 어긋난다. "입문자도 블록 조립하면 쉽다"는 가정은 선례(n8n·Blender Geometry Nodes·Nuke·ComfyUI)에 의해 반복 반박된다 — 강력하지만 가파른 학습곡선이 평균 사용자를 배제한다([medium](https://medium.com/@fadimantium/2026-the-year-of-the-node-based-editor-941f0f15d467)).

## 2. THESIS와의 적합성 — 대체인가, 추가 레이어인가

**대체(replacement)로는 절대 안 되고, 추가 레이어(additional layer)로만 가능하다.** 그것도 입문자 흐름과 물리적으로 분리된 Quant 전용 레이어.

코드 검증으로 확인된 철학 충돌 지점:

- **철학③(입문자에 raw 수치/복잡도 숨김) 정면 위배.** 노드 에디터는 정의상 raw 변수·수식·연결을 사용자 면전에 펼친다. 현 시스템은 `_mask_snapshot_for_free`(routes.py)로 Free에 원시 수치를 마스킹한다 — 노드 캔버스는 이 마스킹을 구조적으로 무력화한다. 보강으로 풀 충돌이 아니라 **제품 정체성의 부정**이다.
- **철학⑥(과신 방지) 구조적 붕괴.** 현 엔진은 coverage<0.5 강제 중립·'매도' 라벨 부재·3중 disclaimer로 과신을 **코드 레벨**에서 막는다. 사용자가 임의 조립한 그래프는 이 가드레일 밖에 있다 — garbage-in→garbage-out 결과를 IKEA 효과("내가 만든 분석이니 맞다")로 과신한다.
- **철학①('사세요' 금지) + 법적 리스크.** 사용자가 만든 노드가 "DCF 적정가 12만원 vs 현재가 8만원 → 매수" 류를 토해내면 한국 유사투자자문/자본시장법 투자권유 오인 리스크(docs/01 §3-11 '치명 우선')를 **사용자 입력 경로로 우회**시킨다. 자유도가 높을수록 가드 강제가 기하급수적으로 어려워진다.

**BM 적합성(중요):** 게이팅 축을 "블록 종류(어떤 블록이 유료냐)"가 아니라 **"블록으로 무엇을 하느냐"**로 옮기면 기존 "결론 무료/근거 유료"(철학⑥/⑦)를 깨지 않고 끼울 수 있다:
- **Free** = 계산된 결론은 종전대로 즉시(에디터 강제 금지). 에디터는 결론 근거를 펼쳐보는 **읽기 전용 뷰**로만.
- **Pro** = 블록 입력값 편집·시나리오 재실행(근거 유료 철학과 정합).
- **Quant** = 블록 저장·공유·자동실행·백테스트/스크리너 연결(자동화·확장).

"블록을 차차 언락"을 무료 유입의 1차 표면으로 쓰면 핵심 미끼(즉시 결론)를 페이월 뒤로 미는 **역행**이 된다 — aha moment 지연으로 전환율 하락([revenuecat](https://www.revenuecat.com/blog/growth/hard-paywall-vs-soft-paywall/)).

## 3. 만약 추진한다면 — 가장 작은 검증용 MVP (스코프 엄격)

**원칙: React Flow를 도입하지 마라. 노드 배선도 만들지 마라.** "조립을 원하는 사용자가 실재하고 유료전환되는가"라는 가설만 가장 싸게 검증한다.

MVP 스코프 (이 범위를 절대 넘지 않는다):
1. **노드/엣지/캔버스 없음.** 대신 docs/01 §3-5에 이미 있는 **Quant 커스텀 스크리너**: 검증된 지표 필터(ROE/PER/영업이익률)를 드롭다운으로 **조합·저장·재실행**. 이것이 "조립 수요"의 80%를 흡수하면서 타입체크·사이클·빈캔버스 문제를 전부 회피한다.
2. **임의 수식 입력 금지.** 사용자는 THESIS가 이미 검증한 21개 지표 사전의 블록만 고른다. garbage-in을 원천 차단.
3. **자작 매수신호/DCF '적정가 버튼' 금지.** 결과는 "이런 근거들이 있습니다" 프레이밍으로만(철학①). Quantopian 교훈(아마추어 룰 과적합)·자본시장법 회피.
4. **모든 사용자정의 결과물에 disclaimer + coverage 가드 엔진단 강제 주입.** ("이 분석은 귀하가 구성했으며 THESIS의 검증을 거치지 않았습니다.")
5. **Quant 티어 전용.** 입문자 대시보드(RegimeSignalCard 구조)는 불변.

이 좁은 MVP에서 "프리셋 조합을 저장·재실행하는 유료 사용자가 실제로 붙는다"가 확인되면, **그 다음에야** 비로소 노드 캔버스가 자연스러운 UI 진화가 된다. 캔버스는 시작점이 아니라 종착점이다 — Composer가 실제로 택한 경로(템플릿 → 빌더).

## 4. 데이터 선행조건 (이게 없으면 전부 무의미)

**코드 직접 검증 결과 — 블록에 연결할 데이터가 현재 0%다.** 이게 이 아이디어의 진짜 80% 작업이고, 가장 과소평가된 비용이다.

- `backend/app/data/collector.py` → `_extract_closes`는 `hist["Close"]` **종가 시계열만** 추출한다. 손익계산서·재무상태표·현금흐름표는 **수집하지 않는다.**
- `backend/app/tickers.py` → 16티커는 전부 가격/지수. L4 4개사(NVDA/AVGO/삼성/하이닉스)도 "수집되나 regime 미채점"(docs/01 §3-4).
- 제안의 핵심 예시 블록(ROE=순이익/자본, PER, PEG, 영업이익률, DCF)은 **전부 재무제표 필요** → 현재 backend에 ROE/PER/fundamentals 엔진 부재.
- docs/01: 기업 펀더멘털 카드·21지표 사전·목표주가·켈리계산기 **전부 '미래(데이터 부재)'**, 의존성 "데이터 적재 / 기업데이터 소스".

선행조건(순서대로):
1. **다년 재무제표 파이프라인 신설** — 수집·정규화·통화/기간/단위 정합·신뢰도 메타·결측 추적. 이게 노드 에디터의 90% 선행조건이고, 안 만들면 스크리너·기업분석(L4)도 못 만든다.
2. **데이터 신뢰성 해결.** yfinance 펀더멘털은 문서화된 결함 — 전 티커·전 재무제표 라인아이템 **1년 컬럼 밀림**([issue #2584](https://github.com/ranaroussi/yfinance/issues/2584), 'not planned'로 종료=미수정), 일부 티커 **빈 DataFrame**([issue #191](https://github.com/ranaroussi/yfinance/issues/191)). 한국 종목 커버리지는 더 얇다. 해결 못 하면 노드 에디터는 **"정밀해 보이는 틀린 결과 생성기"**가 되어 철학⑤·⑥을 사용자 손으로 위반시킨다.
3. **한국 공식 소스 검토** — DART OpenAPI(전자공시) 직접 파싱 등. yfinance만으로는 "짠 나온다"가 한국 종목에서 "짠 안 나온다"가 된다.

**데이터 토대 없이 캔버스 UI부터 시작하는 것은 거꾸로 된 순서다.**

## 5. 예상 공수 / 복잡도

| 구성요소 | 복잡도 | 비고 |
|---|---|---|
| 다년 재무제표 데이터 파이프라인 | **매우 높음** | 진짜 80% 작업. 노드 에디터 없이도 스크리너/L4에 어차피 필요 |
| 데이터 신뢰성 검증·결측 메타 | **높음** | yfinance 밀림·한국 커버리지 해결(유료소스/DART) |
| 블록 스키마·타입시스템(스칼라/시계열/재무제표테이블) | **높음** | 단위·통화·기간 정합성 |
| DAG 평가엔진(위상정렬·사이클검출·NaN 전파·부분 재계산) | **높음** | 기존 regime 엔진이 '고정 DAG'라 개념 토대는 있으나 범용화가 어려움 |
| 포트 타입체크·에러상태 UX | **중간** | "ROE 블록에 가격 시계열 꽂으면 거부" |
| React Flow 캔버스 껍데기 | **낮음~중간** | OSS 성숙([Computing Flows](https://reactflow.dev/learn/advanced-use/computing-flows)). 스택(Next+TS)과 충돌 없음 |
| 그래프 직렬화/버전관리/저장 | **중간** | Supabase 스키마 신설 |
| tier별 블록 카탈로그 게이팅 | **낮음** | 기존 `profiles.tier` 서버 도출 게이팅 재사용 — `required_tier` 필드 1개 추가 |

**역설:** 동기는 "복잡도 위임"이었으나, 노드 에디터는 기능 N개를 안 만드는 대신 **"임의 N개 조합을 안전하게 평가하는 범용 엔진"**을 만들어야 한다. React Flow가 UI 껍데기는 줄여주지만 진짜 비용(데이터 파이프라인·타입세이프 평가엔진·신뢰성 검증)은 그대로 남고 오히려 범용화 때문에 커진다. **복잡도를 줄이는 게 아니라 한 자릿수 배수로 늘리고, "관측 가능한 복잡도(고정 화면 5개)"를 "관측 불가능한 복잡도(임의 사용자 그래프)"로 바꾼다.** 고정 화면 디버그보다 임의 그래프의 "왜 이 숫자가 NaN이냐"가 훨씬 어렵다.

## 6. 최상위 리스크와 디리스킹

| # | 리스크 | 심각도 | 디리스킹 |
|---|---|---|---|

| R3 | 가짜 정밀도 — yfinance 1년 밀림·한국 커버리지로 틀린 DCF 양산 → 신뢰(유일 차별점) 직격 | 치명 | 신뢰도 메타·결측 표면화. 정밀 의존 블록(DCF·다년 현금흐름)은 데이터 정밀도 확보 전 **유료화 금지** |

| R5 | 스코프 크리프 — 미완 5단계 건너뛰고 가장 무거운 인프라 먼저 | 높음 | 출시 전 필수(마이그레이션·E2E 인증·법적고지) + 5단계(스크리너·포트폴리오) 완료 후 6단계 이후로 명시 후순위 |

| R7 | 유지보수 폭증 — 사용자 그래프 = 무한 버그 리포트 | 중간 | 프리셋 조합으로 조합 공간 제한. 자유 배선은 최후순위 |

## 7. 사용자에게 물을 결정사항

1. **타겟 우선순위 확정** — 이 기능을 위해 입문자(다수 유입층)를 포기할 의향이 있는가? 아니면 "Quant 전용 부가 레이어"로 명확히 못 박는가? (이 답이 GO/PIVOT/NO-GO를 가른다.)
2. **데이터 투자 의향** — 노드 에디터 이전에 **다년 재무제표 데이터 파이프라인**(한국 DART 포함)을 먼저 구축하는 데 수개월을 쓸 수 있는가? 못 쓴다면 이 아이디어는 전제부터 성립 불가다.
3. **"조립 수요"의 실증 데이터가 있는가?** — "사용자들이 자꾸 원한다"는 정성 신호인가, 실제 사용자 요청/이탈 로그인가? 먼저 좁은 커스텀 스크리너로 가설검증 후 확장하는 단계적 경로에 동의하는가?
4. **법적 가드 비협상 요건 수용** — 자작 결과물에 강제 disclaimer + coverage 가드 + "매수 적정가 버튼 금지"를 비협상 요건으로 받아들이는가?
5. **순서 동의** — MVP 5단계와 출시 전 필수부터 끝낸 뒤(6단계 이후) 착수에 동의하는가, 아니면 지금 당장을 고집하는가?

## 8. 핵심 통찰 — "사용자 자가조립으로 구현복잡도를 줄인다"는 전제가 맞는가

**틀렸다 — 정확히 역설이다.**

동기는 정당하다. "사용자가 내가 안 만든 조합을 자꾸 원한다 → 일일이 구현하면 복잡도 폭증"은 실재하는 long-tail PMF 신호이고, 조합을 사용자에게 위임하려는 발상 자체는 타당한 플랫폼 전략이다.

그러나 **"자가조립 = 구현 복잡도 감소"라는 인과는 성립하지 않는다.** 기능 N개를 안 만드는 대신, "임의의 N개 조합을 **안전하게** 평가하는 범용 엔진"을 만들어야 하기 때문이다. 그 범용 엔진(블록 타입시스템 + DAG 평가기 + NaN 전파 + 신뢰성 검증 + 에러 UX + 그래프 영속화)은 고정 기능 5~10개를 만드는 것보다 명백히 더 크다. 게다가 데이터 파이프라인이라는 진짜 비용은 어느 쪽이든 똑같이 든다.

즉 **"조합 수요 위임"의 대가로 "플랫폼 빌딩 + 관측 불가능한 복잡도 + 무한 지원 케이스"라는 더 큰 복잡도를 떠안는 거래**다. 복잡도를 줄이는 게 아니라, 위치를 옮기고(개발→유지보수) 총량을 키우는 것이다.

**올바른 해법:** "복잡도 위임"이라는 진짜 동기는 노드 에디터가 아니라 **더 싼 수단**으로 먼저 해결하라 — 검증된 21개 지표 블록의 **제한된 프리셋 조합**(커스텀 스크리너 저장). 이것이 조합 수요의 대부분을 흡수하면서 타입체크·사이클·빈캔버스·가짜정밀도·법적리스크를 전부 회피한다. **무한 캔버스가 아니라 검증된 블록의 제한된 조합으로 좁히면, 철학을 지키면서 수요의 80%를 흡수할 수 있다.** Stacklands의 "단일 규칙·발견의 재미"는 차용하되, n8n의 "빈 캔버스·무한 자유도"는 입문자에게서 격리하라.
