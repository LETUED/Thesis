[사이클 6/6 · 리뷰] 독립 평가자로 검토하고, 발견 즉시 고친다. 사람 개입 없음.

집중할 부분(선택): $ARGUMENTS

독립 평가 원칙(핵심): 자기가 쓴 코드를 자기가 통과시키지 않는다. 검토는 구현과 분리된 fresh-context 평가자에게 맡긴다 — code-reviewer 서브에이전트를 zero-context로 띄우거나(가능하면 다른 모델) /code-review 를 쓴다. "작성자가 쓴 테스트는 필요하나 불충분"이기 때문(테스트 통과 ≠ 기능 정상).

검토할 것:
- 정확성 — 버그, 엣지케이스, 기능 후퇴(regression)
- 보안 — injection, 시크릿 노출, 인증/게이팅 누락
- 단순화·재사용 — 중복, 죽은 코드, 과한 추상화
- 설계철학 준수 — 결론 먼저, 수치 비노출, 과신방지 문구

검증 원칙: 단일 판정을 믿지 않는다. 구조 게이트(테스트·빌드·린트)와 내용 리뷰는 서로 다른 실패를 잡으니 둘 다 본다. 확신 없는 발견은 보수적으로 처리한다.

선택(철저검토): 변경이 크거나 위험하면 더 깊은 클라우드 멀티에이전트 리뷰 `/code-review ultra` 를 제안한다. 단 이건 사용자 트리거·과금이라 Claude가 직접 못 띄운다 — cy6은 제안만 하고, 기본 검토는 code-reviewer 서브에이전트 / `/code-review` 로 진행한다.

자율 루프: 발견사항을 직접 수정 → 테스트·빌드 재확인 → 독립 평가자로 다시 검토. 새 발견이 없을 때까지 반복(최대 2라운드).

기록(decision log) — 사람은 루프 '밖'에서 검토하므로 영속 기록을 남긴다:
- `docs/cycles/<YYYY-MM-DD>-<기능슬러그>.md` 에 이번 사이클 기록을 작성한다(디렉토리 없으면 생성). 날짜는 오늘, 슬러그는 작업을 요약한 kebab-case.
- 담을 내용(이번 대화에서 확보한 만큼 — /cy1~5 가 같은 세션에 있었으면 전부, 단독 /cy6 면 리뷰 위주):
  - **작업**: 스코프 1줄 / **브랜치** / **날짜**
  - **조사(cy1)**: 현황 요약 + 범위 가설
  - **검색(cy2)**: 재사용한 것 / 새로 짠 것
  - **설계(cy4)**: 채택한 설계 + 기각한 대안과 이유
  - **갈린 판단**: 판단이 갈렸던 지점 → 택한 기본값 + 이유 (← 안 남기면 휘발되는 핵심)
  - **진행(cy5)**: 변경 파일 목록 + 테스트·빌드 결과
  - **리뷰(cy6)**: 발견사항 + 처리 결과
  - **상태**: 그린/미해결 잔여
- 기존 같은 파일이 있으면 덮어쓰지 말고 회차를 덧붙인다.

종료: 변경 요약 + 처리한 발견 목록 + 기록 파일 경로 출력. 커밋은 자동으로 하지 않는다 — 마지막에 /commit 을 제안만(작업 브랜치 보존). 사이클 종료.
